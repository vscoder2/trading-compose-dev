## Profit-Lock Math for aggr + paper_live_style_optimistic

### Adaptive threshold formula

For day `i` (after warmup), with base threshold `T_base=10%`:

1. Compute annualized realized volatility on adaptive symbol (`TQQQ`) over prior `w=14` closes.
2. Compute ratio: `r = RV / baseline`, baseline is `85%`.
3. Raw threshold: `T_raw = T_base * r`.
4. Clamp: `T_i = clamp(T_raw, 8%, 30%)`.

If insufficient history, threshold uses base value.

Reference:

- Backtest adaptive series: `run_last_6m_cpu_gpu_backtests.py:431-467`
- Runtime adaptive threshold (same concept): `runtime_backtest_parity_loop.py:122-145`

### Trigger and trailing logic

For a held symbol on day `i`:

1. `trigger_price = prev_close * (1 + threshold_i)`
2. Profit-lock arming condition: `day_high >= trigger_price`
3. Trailing mode stop: `trail_stop = day_high * (1 - trail_pct)` with `trail_pct = 2%`
4. Exit condition: `day_close <= trail_stop`

Execution price in replay depends on effective exec model:

- `market_close`: execute at `day_close * (1 - slip)`
- `synthetic` (and thus `paper_live_style_optimistic`): execute at `trail_stop * (1 - slip)` (or fixed-mode trigger for fixed/partial)

## Runtime/Paper-Live Parity Loop Architecture

### Runtime Cycle

| Stage | Runtime behavior |
|---|---|
| Market gate | Skip if market closed; wait for next loop |
| Time gate | Run only at/after `--eval-time` (default `15:55` NY) |
| Once-per-day lock | Skip if already executed today (`parity_executed_day`) |
| Data load | Pull daily bars (for baseline + threshold) and 1-min intraday bars (for day high/last price) |
| Signal build | Build profit-lock signals from position qty + trigger/trailing checks |
| Exit execution | Execute per `--profit-lock-order-type` |
| Rebalance | Build intents toward baseline target and submit market or bracket |
| Event log | Persist cycle events into state store |

References:

- Runtime main loop: `runtime_backtest_parity_loop.py:291-528`
- Rebalance intent builder (sell-before-buy): `execution/orders.py:67-68`

### Runtime order surface vs replay model

| Concern | Runtime (`runtime_backtest_parity_loop.py`) | Backtest replay (`run_last_6m_cpu_gpu_backtests.py`) |
|---|---|---|
| Exit order types | `close_position`, `market_order`, `stop_order`, `trailing_stop` | Not order-book simulated; uses daily fill model |

| Fill granularity | Broker/exchange behavior | Deterministic EOD synthetic/market_close emulation |
| `paper_live_style_optimistic` switch | N/A in runtime CLI; runtime controls are order-type based | Alias to synthetic effective model |

## Why `paper_live_style_optimistic` usually outperforms market-close style in replay

| Mechanism | Effect |
|---|---|
| Uses trigger/trailing level fill instead of close-only fill | Captures intraday gain retention before weak closes |
| Daily high + trailing stop emulation | Gives optimistic but deterministic lock behavior |
| Same logic on CPU and GPU | Enables exact parity comparison |

## Pros and Cons (paper_live_style_optimistic for aggr)

| Type | Item | Detail |
|---|---|---|
| Pro | Higher upside retention in replay | Synthetic trigger/trailing fills preserve intraday lock levels better than close-only exits |
| Pro | Deterministic and reproducible | Same math on every run; easy CPU/GPU parity testing |
| Pro | Tight profile behavior | Aggressive `10/2` lock with adaptive threshold can lock gains quickly in high RV regimes |
| Pro | Clear attribution | Reports store requested and effective exec model fields |
| Con | Optimistic relative to broker reality | Real stop/trailing execution may slip/queue/partial-fill differently |
| Con | Daily bar abstraction | Intraday path-dependency is simplified into high/close checks |
| Con | Potential over-trading in choppy tapes | Tight trailing stop can trigger frequent exits and re-entries |
| Con | Can diverge from paper/live realized returns | Runtime fills depend on order type, timing, spread, and broker behavior |

## Practical Interpretation for Your Use Case

If you run `aggr_adapt_t10_tr2_rv14_b85_m8_M30` with `--profit-lock-exec-model paper_live_style_optimistic` in backtests:

- You are intentionally selecting the optimistic synthetic fill behavior.
- This is expected to be stronger than `market_close` style replay.
- It is best used for research/upper-bound style benchmarking.
- For realistic paper/live expectation setting, compare against runtime order-type simulations and/or `market_close` replay.
